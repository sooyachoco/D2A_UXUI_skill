# 시나리오 작성 워크플로 (Write Scenario)

> **이 문서의 출발점 (왜 필요한가):**
> 선행 작업(0610)에서 시나리오를 아주 구체적으로 잘 썼는데도 메인 기능 화면이 빈 껍데기로 나왔다.
> 두 가지가 겹쳤다 — ① 산문 시나리오가 "무엇을"만 채우고 "어떻게"는 1c가 맡는데 그 경계가 모호했고,
> ② **AI가 기능 흐름을 임의 가정**해 사용자 소유를 침범한 채 코딩으로 굳혔다(`ui-design-workflow §0` 실패 ⑥).
> 이 스킬은 그 둘을 분리·차단한다: 흐름은 사용자가 확정하고, 시나리오는 커버리지까지만 책임진다.

---

## 0. 제1원칙

### P1 — 기능 흐름은 사용자 소유다 (AI 추측 금지)
"사용자가 무슨 순서로 무슨 일을 하나"는 **사용자가 확정한다.** AI는 strawman(가정안)을 펼쳐 *교정받을* 뿐,
혼자 정해 굳히지 않는다. 이걸 어기면 `ui-design-workflow §0` 실패 ⑥(임의 가정→코딩으로 굳힘)이 재발한다.

### P2 — 커버리지는 보장, 충실도는 1c에 위임
시나리오는 **모든 단계가 빠짐없이 있다(무엇을·왜·순서·위계)**까지만 책임진다.
**"각 단계가 화면에서 어떻게 보이고 조작되나(어떻게)"는 여기서 정하지 않는다** — 그건 `ui-design-workflow`(1c)의 일이다.
대신 **메인 기능을 ★로 표시**해 1c가 어디를 깊게 팔지 지정한다(실패 ① 차단 — 최난도를 추상적으로 처리하는 것).

> "잘 쓴 시나리오 ≠ 잘 설계된 화면." 인터랙션을 여기서 과도하게 적으면(over-spec) 1c의 발산을 막고,
> 안 적으면(under-spec) 빈 껍데기가 된다. 정답은 **"여기가 핵심"이라고 표시하되 어떻게는 비워두는 것**.

### P3 — 페르소나·여정은 SSOT에서 읽고 재정의 금지
대상 페르소나·여정은 `refs/ux-research/`의 단일 source를 **read-only로** 쓴다. 이 스킬에서 페르소나를 새로 만들지 않는다.
비어 있으면 🔵 가설로 표기하고 `ux-research-sync`를 권유한다.

## 1. 파이프라인 위치

```
refine-prd / PRD       [이 스킬 · write-scenario]              reference-proposal → ui-design-workflow
(목적+1차 요구)   ──▶   흐름 strawman → 사용자 확정       ──▶   (화면 후보·job)     (흐름·화면)
        +              페르소나별 과업 시나리오 서술
refs/ux-research       잠정 화면 후보 도출
(페르소나·여정 SSOT)
   ↑ read-only(재정의 금지)               ↓ 산출
                              specs/{NNN}/scenario.md
```

- **앞 단계**: `refine-prd` 또는 PRD. 목적+1차 요구가 있어야 한다.
- **뒤 단계**: `reference-proposal`(화면 후보·job을 입력으로) → `ui-design-workflow`(확정된 흐름·화면을 입력으로).

### 흐름 확정 게이트의 소유 (ui-design-workflow와의 분담)

이 스킬이 **흐름 확정(사용자 소유 게이트)을 소유**한다. 확정된 `scenario.md`가 있으면
`ui-design-workflow` STEP 1은 흐름을 **재질문하지 않고** scenario.md를 로드해 `flow.md`로 락만 건다.
(이 분담을 위해 ui-design-workflow STEP 1에 scenario.md 감지 절차가 추가돼 있다.)

### 기존 스킬과의 경계 (중복 방지)

| 대상 | 역할 | 이 스킬과의 관계 |
|---|---|---|
| `refs/ux-research/USER_SCENARIOS.md` (SSOT) | 페르소나 과업 목표·성공조건(상류, UT가 read-only 소비) | **읽어서** *이 기능의* 흐름 시나리오로 구체화. 페르소나·목표 재정의 금지. |
| `ui-design-workflow` STEP 1 | 흐름을 화면으로 내려보냄 + flow.md 락 | 이 스킬이 흐름을 *확정*, 1c는 그걸 *소비*. flow.md는 1c가 쓴다(여기선 안 씀). |
| `ux-audit` | PRD/플로우 UX 리스크 진단(advisory) | 보완 관계 — scenario.md를 ux-audit로 점검 가능. |
| 인터랙션 충실도(어떻게) | — | **이 스킬은 다루지 않는다**(1c 소관). |

## 2. 실행 런북 — 각 스텝: [내 액션] · [사용자 유도] · [게이트]

> **핵심: 사용자가 흐름을 *떠올리게* 하지 말고, 내가 strawman을 *유도*한다. 단 확정은 사용자가 한다.**

### STEP 0 · 입력·페르소나 정렬
- **내 액션**: PRD(목적+1차 요구) + `refs/ux-research/PERSONA.md`·`USER_JOURNEY_MAP.md`·`USER_SCENARIOS.md`를 로드. 이 기능의 대상 페르소나 1~3을 SSOT에서 선정(**재정의 금지**). SSOT가 비었으면 🔵 가설로 표기하고 진행하되 `ux-research-sync` 권유.
- **유도**: "이 기능의 주 사용자는 누구(P1 신규 / P2 헤비 / P3 접근성 중)인가요?"
- **게이트**: 대상 페르소나 + 핵심 목표 확보 → STEP 1

### STEP 1 · job 추출 (동사 + 대상)
- **내 액션**: PRD 기능을 **"[동사] [대상] — [핵심 인터랙션 1줄]"**로 리라이트한다(`ui-design-workflow` §5 표). 명사·모호어("관리/처리")는 매칭 불가 → 동사로 변환. **메인 기능 1개를 ★표시**(이 제품의 차별점·최난도. 로그인·대시보드가 아니다).
- **유도**: "메인 기능은 이것 맞나요? job 표현이 맞나요?"
- **게이트**: job 목록 + ★메인 확정 → STEP 2

### STEP 2 · 흐름 strawman → 사용자 확정 ★(P1의 핵심)
- **내 액션**: 내가 가정한 "사용자 행동 순서"를 strawman 다이어그램(글/화살표)으로 펼친다. **화면은 그리지 않는다(무엇을·순서만).** 분기·예외 경로도 가정안으로 표시.
- **유도**: "이 순서 맞나요? **틀린 화살표·순서·빠진 분기를 고쳐주세요.**" — AI 추측 금지, 사용자 소유.
- **게이트**: **사용자 확정** → 확정 흐름 기록 (실패 ⑥ 차단)

### STEP 3 · 페르소나별 과업 시나리오 서술
- **내 액션**: 확정된 흐름을 페르소나 관점 서사로 푼다. 각 단계마다 **`트리거 → 행동 → 기대 결과 → 성공조건`**. **분기/실패 경로**(예: 권한 없음, 입력 오류, 네트워크 실패 시 무엇을)도 포함. 성공조건은 가능하면 측정 가능하게(ai-usability-test의 과업 성공 기준과 호환).
- **유도**: "각 과업의 성공조건이 이게 맞나요? 실패하면 어떻게 되길 기대하나요?"
- **게이트**: 페르소나별 시나리오 합의 → STEP 4

### STEP 4 · 잠정 화면 후보 도출
- **내 액션**: 확정 흐름에서 **화면 후보를 잠정 나열**하고 각 화면에 job을 매핑한다. **확정·통합·연결은 `ui-design-workflow` STEP 1에 위임**(여기선 후보까지만). 중복 가능 화면은 "통합 후보"로 표시.
- **유도**: "이 화면 후보들에서 빠진 것/합쳐도 될 것 있나요?"
- **게이트**: 화면 후보 + job 매핑 → STEP 5

### STEP 5 · 커버리지 자가점검 (scenario-fidelity 게이트)
- **내 액션**: 시안 제출 전 아래를 **명시 출력**(✅/⚠️). 통과분만 제출.

| # | 항목 | 통과 기준 |
|---|---|---|
| 1 | PRD job 전수 커버 | PRD의 모든 기능 job이 시나리오에 등장 |
| 2 | 페르소나 목표 커버 | 대상 페르소나의 핵심 목표가 과업으로 존재 |
| 3 | 여정 단계 커버 | USER_JOURNEY_MAP 단계(유입~이탈)와 어긋남 없음 |
| 4 | 분기/실패 경로 | 주요 과업마다 실패·예외 경로 1개 이상 |
| 5 | 메인 기능 표시 | ★메인이 정확히 1개 지정 + 흐름상 위치 명확 |
| 6 | 충실도 미침범 | 화면 레이아웃·인터랙션 "어떻게"를 적지 않았다(1c 영역) |

- **게이트**: 6/6 ✅ → STEP 6. ⚠️ 있으면 수정 후 재점검.

### STEP 6 · 핸드오프
- **내 액션**: `specs/{NNN}/scenario.md` 저장(§3 스키마). 활동 로그 기록(§4).
- **유도**: "시나리오 확정했습니다. 이대로 `reference-proposal 실행해줘`로 레퍼런스 제안 갈까요? (또는 바로 `ui-design-workflow`)"
- **게이트**: 파일 존재 → 1a/1c 진입 가능.

## 5. 사이드레일 — "흐름이 자꾸 바뀔 때"
1. **strawman은 1회만 내 가정으로, 이후는 사용자 교정 반영만.** 내가 두 번째 가정안을 또 만들지 않는다(추측 누적 금지).
2. **"어떻게"가 끼어들면 STEP 4로 밀어낸다.** 흐름 논의 중 레이아웃·버튼 위치 얘기가 나오면 "그건 화면 설계(ui-design-workflow)에서"로 명시 분리.
3. **2-strike**: 흐름이 2회 연속 크게 뒤집히면 PRD(목적+1차 요구)가 흔들리는 신호 → `refine-prd`로 후퇴 권유.

## 6. 관련 문서
- `refine-prd` — 앞 단계(목적+1차 요구 정제).
- `refs/ux-research/PERSONA.md`·`USER_JOURNEY_MAP.md`·`USER_SCENARIOS.md` — 페르소나·여정 단일 source(read-only).
- `reference-proposal` — 뒤 단계(화면 후보·job을 받아 레퍼런스 제안).
- `ui-design-workflow` — 뒤 단계(확정 흐름·화면을 받아 "어떻게" 설계). STEP 1이 scenario.md를 소비.
- `ux-audit` — scenario.md/PRD의 UX 리스크 사전 진단(보완).
